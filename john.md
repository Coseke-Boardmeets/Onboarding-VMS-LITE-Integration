# VMS Lite — Visitor Management System Documentation

Welcome to the comprehensive documentation for **VMS Lite**, a Visitor Management System built for receptionist management and visitor registration. This document provides a complete guide to the project’s structure, stack, database schema, API contracts, backend layers, frontend components, configuration, and test suites.

---

## 1. Project Overview

VMS Lite is a full-stack TypeScript application designed with a clean architecture (enforcing SOLID and DRY principles). It facilitates two core flows:
*   **Visitor Registration Flow**: A public interface (`/register`) where visitors or receptionists can pre-register by inputting their full name and the purpose of their visit.
*   **Reception Dashboard Flow**: A secure, authenticated control center (`/`) where receptionists can view a list of all visitors, check them in (transitioning status from `PENDING` to `CHECKED_IN`), check them out (transitioning status to `CHECKED_OUT`), and view real-time statistics/analytics.

## 3. Database Schema

The database uses PostgreSQL, managed via Prisma. The configuration is defined in [`backend/prisma/schema.prisma`](file:///c:/Users/johnk/onboarding-vms/backend/prisma/schema.prisma):

```prisma
generator client {
  provider        = "prisma-client-js"
  output          = "../src/generated/prisma"
  previewFeatures = ["driverAdapters"]
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Visitor {
  id        String   @id @default(uuid())
  fullName  String
  purpose   String
  status    String   @default("PENDING") // PENDING, CHECKED_IN, CHECKED_OUT
  timeIn    DateTime?
  timeOut   DateTime?
  createdAt DateTime @default(now())
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  password  String
  fullName  String
  createdAt DateTime @default(now())
}
```

## 4. API Endpoint Specifications

All data transactions flow through REST endpoints. **Note**: Admin login/authentication is completely bypassed (all actions are publicly accessible and utilize a default System Administrator mock session).

| Method | Route | Description | Auth Required (Original) | Auth Required (Current) | Request Body | Response Format |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **GET** | `/health` | Server status check | No | No | None | `{ "status": "ok" }` |
| **POST** | `/auth/register` | Register a new administrator user | No | No | `{ "email", "password", "fullName" }` | `{ "user": { "id", "email", "fullName" }, "token" }` |
| **POST** | `/auth/login` | Login and fetch token | No | No | `{ "email", "password" }` | `{ "user": { "id", "email", "fullName" }, "token" }` |
| **GET** | `/auth/me` | Fetch active user credentials | **Yes** | **No (Mocked)** | None | `{ "user": { "id", "email", "fullName" } }` |
| **GET** | `/visitors` | Retrieve all registered visitors | **Yes** | **No (Mocked)** | None | `Array<Visitor>` |
| **POST** | `/visitors` | Register new guest (Public Form) | No | No | `{ "fullName", "purpose" }` | Created `Visitor` object |
| **PUT** | `/visitors/:id/checkin` | Check in a visitor (Pending $\rightarrow$ Checked In) | **Yes** | **No (Mocked)** | None | Updated `Visitor` object |
| **PUT** | `/visitors/:id/checkout` | Check out a visitor (Checked In $\rightarrow$ Checked Out) | **Yes** | **No (Mocked)** | None | Updated `Visitor` object |
| **GET** | `/visitors/stats` | Retrieve receptionist panel analytics metrics | **Yes** | **No (Mocked)** | None | Statistics Payload (see details below) |

## 5. Architectural Implementation

### A. Backend SOLID Layers
The backend implements a clear separation of concerns (SOLID principles):
1.  **Server Initialization (`server.ts`)**: Initializes express, hooks middlewares (`cors`, `json`), mounts routers, and runs a bootstrap script to auto-seed a default receptionist (`admin@coseke.com` / `Password123!`) if no users exist.
2.  **Routes (`routes/`)**: Contains pure endpoint declarations and applies route-specific middleware (`requireAuth`, validation schema).
3.  **Middleware (`middleware/`)**: Performs side-checks:
    *   `auth.middleware.ts` acts as a pass-through: it directly populates `req.user` with a mock user payload and proceeds (bypassing JWT verification).
    *   `validation.middleware.ts` validates that fields like `fullName` and `purpose` exist, are non-empty, and that `fullName` has a minimum length of 2 characters.
4.  **Controllers (`controllers/`)**: Handle HTTP parameters and return JSON payloads. Controllers **never** interact with the database directly.
5.  **Services (`services/`)**: Central place for executing Prisma database queries, keeping operations reusable and clean.

### B. Frontend DRY Structure
The frontend dashboard relies on Next.js 16 Client components:
1.  **API Client (`lib/apiClient.ts`)**: Adheres to the DRY principle. Centralizes `fetch` configurations and error propagation.
2.  **Auth Provider (`components/AuthContext.tsx`)**: Bypassed to immediately return a logged-in `System Administrator` session and ignore login/logout trigger commands.
3.  **Main Page (`app/page.tsx`)**: Integrates real-time search filtering, status switches, automated dashboard refresh, and premium SVG rendering for visitor traffic analytics (including interactive hover states for charts).
4.  **Visitor Table (`components/VisitorTable.tsx`)**: An interactive dashboard showing visitor info, purposes, times, and actionable buttons depending on status (e.g. Check In/Check Out buttons are conditionally displayed).

## 6. Setup and Running the Project

### A. Prerequisites
Ensure you have the following installed:
*   Node.js (v20 LTS or newer)
*   PostgreSQL Database running locally or hosted

### B. Backend Setup
1.  Navigate to the backend directory:
    ```bash
    cd backend
    ```
2.  Install dependencies:
    ```bash
    npm install
    ```
3.  Create a `.env` file from the template:
    ```bash
    cp .env.example .env
    ```
    *Ensure `DATABASE_URL` is set to your PostgreSQL instance (e.g., `postgresql://user:pass@localhost:5432/vms`).*
4.  Push the database schema:
    ```bash
    npx prisma db push
    ```
5.  Start the dev server:
    ```bash
    npm run dev
    ```
    *The API will start at **http://localhost:4000**.*

### C. Frontend Setup
1.  Navigate to the frontend directory:
    ```bash
    cd frontend
    ```
2.  Install dependencies:
    ```bash
    npm install
    ```
3.  Set up Shadcn UI component preferences:
    ```bash
    npx shadcn@latest init
    ```
4.  Create `.env.local` configuration:
    ```bash
    cp .env.example .env.local
    ```
    *During development, the default endpoint points to the local mock `json-server` on port `5000`. Update to `http://localhost:4000` once integrating with the real Express backend.*
5.  Launch the Mock Server (for independent frontend testing):
    ```bash
    npx json-server --watch db.json --port 5000
    ```
6.  Start Next.js dev server:
    ```bash
    npm run dev
    ```
    *The app will be available at **http://localhost:3000**.*
